# Post-Implementation Skill Update

実装が完了したセッションの終わりに、スキルエコシステムを育てるためのレビューと改善を行う。

## 目的

スキルは「一度作ったら終わり」ではなく、実際の使用経験から育てるもの。このスキルは、各実装セッションを「スキル改善のフィードバックループ」に変える。

---

## Step 1: セッションのスキル使用を棚卸し

今回のセッションで何をしたか振り返る。会話履歴をざっと確認し、以下を特定する：

**使用したスキル:**
- `Skill` ツールで明示的に呼んだスキル名をリストアップ
- 発動したスキルの内容が適切に機能したか評価（◎/○/△）

**使えばよかったスキル（使い忘れ）:**
次のスキルと、今回の作業内容を照合する：

.claude/skills/**/SKILL.md

見逃したスキルがあれば、**なぜ見逃したか**を分析する：
- トリガー条件の言葉が足りなかった
- description が曖昧だった
- スキルの存在自体を知らなかった（新規スキルなら記録のみ）

---

## Step 2: 改善が必要なスキルを特定する

以下の基準で改善優先度を判断する：

**高優先度（すぐ直す）:**
- 使い忘れが起きた → description のトリガーワードを強化
- 使ったが機能が不足していた → 内容を補強
- 使ったが手順が古くなっていた → 更新が必要

**低優先度（記録のみ）:**
- 問題なく機能した
- 今回初めて使って問題なし

---

## Step 3: スキルを更新する

### description のトリガーを強化する場合

対象スキルの `SKILL.md` の `description:` フィールドを直接編集する。

トリガーワードを追加するときのポイント：
- ユーザーが**実際に使った言葉**を追加する（抽象的なキーワードより具体的な文脈）
- 「〜するとき」「〜と言われたら」の形で書く
- 過去に見逃した具体的なフレーズを入れる

例：
```
# 変更前
description: Rust コードのテスト方法

# 変更後  
description: Rust コードのテスト方法。「cargo test」「テストが落ちた」「Rust のテスト書いて」「integration test」「backend_integration」と言われたら起動する。
```

### スキルの本文を更新する場合

内容が不足・古くなっていた箇所を修正する。大きな変更が必要なら `skill-creator` を呼び出す。

---

## Step 4: 出力サマリーを提示する

以下のフォーマットで報告する：

```
## スキル振り返りレポート

### 使用したスキル
- skill-name ◎ — 問題なし
- skill-name ○ — 機能したが XX が不足
- skill-name △ — 改善が必要

### 使えばよかったスキル
- skill-name — 理由: [なぜ見逃したか]
  → 対処: [description に追加したキーワード]

### 更新したスキル
- skill-name: [変更内容の1行サマリー]

### 次セッションへの申し送り
- [気づいた点があれば]
```

---

## Notes

- スキルの `name` フィールドとディレクトリ名は**変更しない**
- description 改善は Edit ツールで直接行う。大規模な書き直しが必要なときだけ `skill-creator` を呼ぶ
- linksee-memory を使っているプロジェクトなら、重要な学びを `remember` しておく
